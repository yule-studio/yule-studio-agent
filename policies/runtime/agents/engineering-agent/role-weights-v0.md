# Engineering Agent Role-Default Model Weights v0

## Principle: Role과 Model은 분리된다

이 부서의 핵심 원칙 중 하나는 **역할(role)과 모델(model)을 분리**한다는 것이다.

- **역할**은 책임 단위다 — tech-lead, ai-engineer, product-designer, backend-engineer, frontend-engineer, qa-engineer.
- **모델**은 실행 백엔드 풀이다 — Claude, Codex, Gemini, Ollama, GitHub Copilot. 부서 단위 `participants`/`integrations`로 관리.

따라서 한 역할이 항상 같은 모델을 쓰는 게 아니라, **작업 성격에 맞는 모델을 게이트웨이가 선택**한다. 이때 역할별 기본 가중치(이 문서)와 작업별 보정(다음 단계 디스패처)이 결합되어 최종 모델이 결정된다.

## v0 가중치 설정

다음 표는 **기본값**이다. 작업의 컨텍스트(파일 길이, 보안성, 비용 민감도)에 따라 디스패처가 보정한다.

각 행은 모델 후보 목록이고, 점수가 높을수록 우선 호출된다. 점수 0이면 후보에서 제외된다.

### tech-lead (작업 분해, 의존 순서 결정, 합의 조율)

| 모델 | 가중치 | 메모 |
|---|---|---|
| `claude` | 9 | 추론과 합의 조율에 강점. 기본 추천. |
| `gemini` | 7 | 긴 컨텍스트 분석(여러 이슈 동시 검토)에 유리. |
| `codex` | 5 | 분배 결정 자체엔 약하지만 코드 영향도 평가에 보조. |
| `ollama` | 3 | 로컬 사고/분류용. 비용 민감 환경에서 fallback. |

## 외부 신호 처리 정책

- 외부 모델 평가/리더보드(LMSys, HumanEval 등)는 **참고 신호**일 뿐이며, MVP 단계에서는 **가중치를 자동 갱신하지 않는다**.
- 가중치 변경은 사용자 또는 부서 정책 변경(이 문서) 단위로만 일어난다.
- 미래에는 로컬 성과(승인률, 회귀 발생률)를 약하게 반영하는 메커니즘을 도입할 수 있지만, 그 결정은 별도 정책 문서에서 내린다.

## 작업별 보정 (다음 단계 디스패처가 적용)

게이트웨이/디스패처가 다음 컨텍스트에 따라 위 가중치에 보정을 적용할 수 있다 (구현은 다음 마일스톤):

| 컨텍스트 | 보정 |
|---|---|
| 긴 파일/멀티 파일 분석 | `gemini` +2 (long context) |
| 비용 민감 (CI 등) | `ollama` +3, `claude`/`codex` -2 |
| 보안 민감 (인증, 토큰) | `claude` +2, `ollama` -3 (로컬 모델은 보안 결정에 약함) |
| 빠른 응답 우선 | `codex` +1, `gemini` +1 |
| 멀티모달 (이미지 시안) | `gemini` +5, 그 외 -3 |

## 변경 절차

가중치를 변경하려면:

1. 변경 이유 (어느 작업에서 어느 모델이 좋았는지/약했는지) PR 본문에 명시.
2. 변경 대상 역할만 수정. 다른 역할은 영향 받지 않게.
3. 디스패처 코드(다음 단계)가 이 문서를 source-of-truth로 참조하므로, 디스패처 구현 후엔 PR에 디스패처 입력 데이터도 함께 갱신.
