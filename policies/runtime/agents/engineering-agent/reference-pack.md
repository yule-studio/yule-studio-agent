# Engineering Agent Reference Pack

## 언제 적용되나

작업이 다음 영역과 연결될 때 **구현 시작 전에** 레퍼런스 조사를 먼저 수행한다:

- UI / UX
- 랜딩페이지 / 마케팅 페이지
- 브랜딩 / 시각 톤
- 온보딩 흐름
- 광고 / 프로모션 / 콘텐츠
- 이메일 템플릿
- 사용자 플로우 / 인터랙션 / 화면 전환

위 영역과 연결되지 않는 순수 백엔드 또는 인프라 작업은 레퍼런스 조사를 강제하지 않는다.

## 레퍼런스 우선순위

1. **사용자 제공 링크/스크린샷** — 가장 우선. 사용자가 명시적으로 보여준 자료가 있으면 그것이 1순위 기준.
2. **공식 트렌드/광고/학습 소스** — Pinterest Trends, Canva Design School, Notefolio, Behance 등.
3. **실제 출시된 제품의 UI/플로우 레퍼런스** — Mobbin, Page Flows, Awwwards, Wix Templates, Really Good Emails 등.
4. **지역 감성 참고용 디자인 커뮤니티 레퍼런스** — Notefolio (한국), 지역 커뮤니티 갤러리 등.

## 권장 수량과 산출

가능하면 **최소 3개, 이상적으로 5개 정도**의 레퍼런스를 확보한다.

각 레퍼런스마다 다음 4가지를 명문화한다:

1. **무엇을 참고했는지** (출처, 제품/콘텐츠 이름, 핵심 패턴 한 줄 요약)
2. **어떤 패턴을 차용할지** (현재 작업에 옮겨올 요소)
3. **어떤 요소는 피할지** (저작권 위험, 컨벤션 어긋남, 사용자 맥락 부적합 등)
4. **현재 작업에 어떻게 재구성해 반영할지** (단순 복제 X, 맥락에 맞게 재설계)

## 단순 복제 금지

레이아웃, 카피, 색상, 상호작용을 그대로 가져오는 것은 금지한다. 모든 차용은 현재 프로젝트의 톤, 사용자 맥락, 기능 제약에 맞게 재설계되어야 한다. 위반 시 product-designer가 게이트웨이에 차단 신호를 보내고 재작업 요청한다.

## 역할별 Reference Pack (1순위 소스)

각 역할이 작업 시작 시 가장 먼저 참고하는 소스 모음이다. 모든 역할이 모든 소스를 참고할 수 있지만, 1순위가 우선이다.

### product-designer
**핵심 책임**: 화면 구조, 컴포넌트 분해, 시각 톤 결정.

| 소스 | 활용 |
|---|---|
| **Pinterest Trends** | 시즌별 시각 트렌드, 컬러/타이포 분위기 |
| **Notefolio** | 한국 디자이너 포트폴리오, 지역 감성 참고 |
| **Behance** | 글로벌 프로페셔널 디자인 시스템, 포트폴리오 |
| **Canva Design School** | 마케팅 비주얼/문서/SNS 그래픽 패턴 |
| **Mobbin** | 모바일 앱 실제 화면/플로우 레퍼런스 |

추가 가능 소스: Awwwards (웹 사이트), Wix Templates (랜딩 패턴), Really Good Emails (이메일).

### frontend-engineer
**핵심 책임**: 디자인 결정을 코드로 옮기고 UI 인터랙션 품질 챙기기.

| 소스 | 활용 |
|---|---|
| **Mobbin** | 모바일 앱 실제 화면/패턴 (구현 가능성 검증) |
| **Page Flows** | 사용자 플로우 캡처 (전환 흐름 학습) |
| **Awwwards** | 인터랙션·애니메이션 레퍼런스, 인디 사이트 빌드 패턴 |
| **Wix Templates** | 랜딩 페이지 구성 패턴, 섹션 분해 |

추가 가능 소스: Behance (시안 해석), Really Good Emails (HTML 이메일 패턴).

### tech-lead
**핵심 책임**: 작업 분해와 경쟁사례/공식 문서 기반 의사결정.

| 소스 | 활용 |
|---|---|
| **Page Flows** | 경쟁/유사 제품의 플로우 캡처 (의사결정 근거) |
| **Google Trends** | 시장 신호, 키워드 추이 (우선순위 보정) |
| **공식 문서 / RFC / 프레임워크 가이드** | 기술 결정의 근거 (예: React docs, Next.js, FastAPI) |
| **경쟁 제품 사례** | 사용자 제공 링크, 공개 사례 분석 |

추가 가능 소스: 사용자 제공 자료 (가장 우선).

### qa-engineer
**핵심 책임**: 수용 기준, 회귀 시나리오, 실패 케이스 발굴.

| 소스 | 활용 |
|---|---|
| **Page Flows** | 실제 제품의 빈/로딩/에러 케이스 패턴 학습 |
| **실제 출시된 제품의 사용자 플로우** | 회귀 시나리오 발굴 |
| **사용자 제보 패턴** (있으면) | 자주 깨지는 영역 우선 검증 |

추가 가능 소스: Mobbin (실제 화면의 빈/로딩 상태), 공식 접근성 가이드(WCAG).

### backend-engineer
백엔드는 시각 레퍼런스보다 **공식 문서, RFC, 보안 가이드**가 우선이다. UI 영역과 직접 연결되는 작업(API 응답 모양 → 화면 영향)에 한해 product-designer/frontend-engineer가 정리한 레퍼런스 요약을 그대로 받는다.

## 자동 수집 정책

- **Notefolio, Behance** 등은 약관상 **자동 대량 스크래핑 금지**. 수동 참고하거나 사용자가 제공하는 링크 중심으로 본다.
- **Mobbin, Page Flows, Awwwards** 등은 자체 검색이 허용되지만, 봇이 헤드리스 브라우저로 무리하게 접근하지 않는다. 검색 결과 페이지의 메타데이터까지만 사용한다.
- 웹 접근이 불가능한 환경(오프라인, 회사 방화벽 안)에서는 **사용자 제공 자료와 로컬 문서**만 사용하고, 어떤 외부 레퍼런스가 있으면 더 좋아질지 PR 본문에 메모한다.

## 산출 양식 (PR/이슈에 포함)

레퍼런스 조사 결과는 다음 양식으로 PR 본문이나 이슈 코멘트에 첨부한다:

```markdown
## References Surveyed
1. [출처 1](URL or 제목) — 차용: <한 줄> / 회피: <한 줄> / 재구성: <한 줄>
2. [출처 2] ...
3. [출처 3] ...
(이상 5개)

## What This Implementation Took From References
- <간단한 요약>

## What This Implementation Deliberately Avoided
- <간단한 요약>
```

이 양식이 있으면 다른 멤버(특히 product-designer, qa-engineer)가 후속 작업에서 같은 레퍼런스를 재활용하거나 검증 기준으로 사용할 수 있다.
